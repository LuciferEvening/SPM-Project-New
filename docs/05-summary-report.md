## 二、WBS分解结果及其三级分解逻辑

### 2.1 分解原则
1. **结果导向**：所有条目均为可交付成果，不以纯动作作为分解主线
2. **三级分解**：一级为成果域，二级为子成果，三级为最小成果单元
3. **完整性**：覆盖项目全部工作范围
4. **独立性**：各交付物之间相对独立，避免交叉重叠

### 2.2 WBS分解结果

**一级交付物**：
1. 项目管理成果
2. 业务功能成果
3. 技术实现成果
4. 测试验证成果

**二级交付物**（部分示例）：
- 1.1 启动成果
- 1.2 计划成果
- 1.3 控制成果
- 2.1 活动管理功能
- 2.2 报名管理功能
- 2.3 签到管理功能

**三级交付物**（部分示例）：
- 1.1.1 项目章程
- 1.1.2 团队组建方案
- 1.2.1 WBS文档
- 1.2.2 进度计划
- 1.2.3 配置管理计划
- 2.1.1 活动发布模块
- 2.1.2 活动维护模块
- 2.2.1 报名提交模块
- 2.2.2 名单审核模块
- 2.3.1 二维码签到模块
- 2.3.2 签到统计模块

### 2.3 分解过程说明
1. **识别核心成果域**：首先确定项目的四大核心成果域（项目管理、业务功能、技术实现、测试验证）
2. **分解子成果**：每个成果域下进一步分解为具体的子成果
3. **细化交付物**：将子成果分解为可管理的最小成果单元
4. **验证完整性**：检查是否覆盖项目全部工作范围

### 4.2 分支策略

```
main (主分支，保护分支)
  ↓
dev (开发集成分支)
  ↓
feature-wbs (WBS功能分支)
feature-summary (总结报告分支)
feature-wanghanyin (组员工作分支)
```

**分支说明**：
- **main**：主分支，存放稳定的最终版本，仅接受来自dev的合并
- **dev**：开发集成分支，用于整合各feature分支的工作
- **feature-***：功能特性分支，每位成员在各自的feature分支上开发

### 4.3 提交命名规范

采用约定式提交（Conventional Commits）：

| 类型 | 说明 | 示例 |
|---|---|---|
| `init` | 初始化项目 | init: 初始化项目结构 |
| `feat` | 添加新功能/内容 | feat: 完成WBS分解 |
| `fix` | 修复问题/修改内容 | fix: 修正进度计划日期 |
| `docs` | 文档更新 | docs: 更新总结报告 |
| `refactor` | 重构 | refactor: 优化文档结构 |
| `test` | 测试相关 | test: 添加测试用例 |
| `merge` | 分支合并 | merge: 合并feature到dev |

### 4.4 合并规则

**合并流程**：
1. 成员在各自的feature分支完成工作后提交
2. 将feature分支推送到远程仓库
3. 发起Pull Request到dev分支
4. 配置管理员审核后合并到dev
5. 阶段性完成后，由项目经理合并dev到main

**主分支保护规则**：
- main分支设置为保护分支
- 禁止直接push到main
- 必须通过Pull Request合并
- 需要至少一位审核者批准

## 五、冲突制造、识别与解决过程

### 5.1 冲突场景设计

**冲突文件**：`docs/conflict.md`

**冲突内容**：两位成员对代码合并流程有不同意见：
- **成员A（王其潇）**：主张直接合并到main分支，提高效率
- **成员B（王汉寅）**：主张先经过dev再合并到main，保证质量

### 5.2 冲突制造过程

1. **创建冲突场景**：在`docs/conflict.md`文件中设置冲突点
2. **成员A操作**：在`feature-summary`分支修改为"直接合并到main"
3. **成员B操作**：在`dev`分支修改为"先经过dev再合并到main"
4. **触发冲突**：将`feature-summary`合并到`dev`时产生冲突

### 5.3 冲突识别

```
<<<<<<< HEAD
当前结论：本组决定直接合并到 main。
=======
当前结论：本组决定先经过 dev 再合并到 main。
>>>>>>> feature-summary
```

### 5.4 冲突解决过程

1. **识别冲突**：Git自动检测到同一文件同一位置存在不同修改
2. **协商讨论**：团队通过微信群讨论两种方案的优缺点
3. **达成共识**：
   - 直接合并优点：效率高、流程简单
   - 间接合并优点：质量可控、便于集成测试
   - 最终决定：采用"先经过dev再合并到main"的方式
4. **手工解决**：编辑冲突文件，保留协商后的正确内容
5. **提交合并**：完成冲突解决并提交

### 5.5 解决结果

```
当前结论：本组决定先经过 dev 再合并到 main。
```

**理由说明**：经过团队讨论，认为质量保证比开发效率更重要，采用dev分支作为集成分支可以更好地控制代码质量，便于进行集成测试和代码审查。

- 结果导向的重要性

配置管理实践让我们掌握了：
- 分支策略的设计
- 提交规范的重要性
- 合并流程的控制

### 6.5 冲突解决体会

冲突解决过程中，我们体会到：
- 团队沟通的重要性
- 冲突解决的流程
- 版本控制的优势
